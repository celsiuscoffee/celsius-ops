## 1. Headline findings (prioritized)

### P1 — Attendance AI processing has been dead since 2026-04-19
`hr_attendance_logs`: 290/840 rows ever processed; `ai_processed_at` max =
2026-04-19. Last 7 days: 142/173 logs sit `ai_status='pending'`,
`final_status=null`. The processor (`lib/hr/agents/attendance-processor.ts`)
is only invoked from a manual button on the HR dashboard
(`(admin)/hr/page.tsx:132` → `POST /api/hr/attendance/process`) — **no cron
runs it**. Base pay is safe (regular/total hours are computed at clock-out;
168/173 recent logs populated, and the monthly calculator's NULL-safe filter
pays non-rejected logs), but OT is not: see P2.

### P2 — All July OT (160 h) is currently unpayable; 141 OT requests pending
The monthly calculator only pays OT when a log passes `isOtApproved`
(`payroll-calculator.ts:293-296`: manager-approved/adjusted, or AI-approved).
With the processor dead (P1) and the OT-request queue unworked
(`hr_overtime_requests`: 141 pending — 81 from April, 60 from July), **every
July OT hour fails that check**. The `overtime-requests/sync` cron dutifully
creates pending requests nightly; nobody approves them. If a payroll run is
computed today, OT pays RM 0. Human action: work the OT queue (or decide the
April backlog is dead and bulk-reject it); candidate for the OT-approval
chase loop (§4).

### P3 — PCB under-withholding: EPF relief capped at RM 7,000 instead of RM 4,000
`statutory/calculators.ts:206-207` applies
`min(annualEpfContribution, EPF_CAP=7000)` — but RM 7,000 is LHDN's
**combined EPF (4,000) + life insurance (3,000)** cap, and the code never
adds life-insurance premiums anywhere (the `life_insurance_relief` profile
column is unread). Anyone earning > ~RM 3,030/mo (annual EPF > 4,000) gets
up to RM 3,000/yr excess relief → MTD under-deducted by ~RM 50-60/mo at
mid brackets. Fix is one line + seed note, but it changes net pay →
**hard rule 6: propose, show diff, get approval before shipping.**
(The live seed row's own note says "Split: life 3000 + EPF 4000".)

### P4 — Payslip distribution layer is a dead end
`hr_payslips` exists in prod (0 rows ever) but **no code reads or writes
it** — no Cloudinary upload, no `viewed_at` stamping, no new-payslip badge.
Staff payslip pages (web + native) exist but are gated off behind
`PAYROLL_UI_ENABLED` ("coming soon") and read `hr_payroll_items` directly.
Spec §2/§8/§9 distribution is NOT BUILT. Decide: build the persistence layer
or delete the dead table + spec sections.

### P5 — Weekly PT calculator ignores OT multipliers and statutory toggle
`payroll-calculator-weekly.ts` pays flat `hourly_rate × all clocked hours` —
no 1.5×/2×/3× OT premium, no statutory even when
`statutory_applicable=true`. The migration's OT-multiplier +
`statutory_applicable` columns are dead for PT. Mitigating reality: PT wages
currently bypass payroll runs entirely (weekly manual bank transfers →
`partimer` GL rule, per STATE.md), so nothing wrong is being *paid* — but
the `/hr/payroll/weekly` page presents numbers a human could act on.
Label it or finish it.

## 4. What can become a loop (ranked)

The repo's loop shape: **sense** (cheap DB observation, usually a cron) →
**agent action** (draft / classify / auto-handle the clear cases) → **human
gate** (hard rule 6 for anything touching pay) → **watchdog digest**
(fingerprint-deduped WhatsApp via ops-pulse, like
`lib/inventory/loop-watchdog.ts`). Candidates, ranked by value:

1. **Monthly payroll close loop** (highest value, rule-6 gated) — auto-run
   preflight + compute into a draft each period, attach anomaly/variance
   digest, chase the approver until confirmed, verify payslips/bank file
   followed; escalate stuck drafts. Blocks on nothing — all building blocks
   exist (`payroll-calculator.ts`, `preflight`, `anomalies.ts`).
2. **OT approval chase** (unblocks P2, no money write) — nightly digest of
   pending `hr_overtime_requests` older than N days to the right manager,
   with one-tap approve/reject links; escalate to owner weekly. The sync
   cron already creates the queue; nothing chases it.
3. **Attendance processing** (fixes P1) — fold `processAttendance()` into an
   existing cron tick (ops-nudges dispatcher or attendance-auto-close;
   **do not add a 41st vercel cron** — budget is 38) so logs stop rotting
   at `ai_status='pending'`.
4. **Weekly PT payroll loop** (rule-6 gated) — Monday auto-compute of last
   week's clock-based PT gross + bank file + reconciliation against the
   `partimer` GL lines; human approves the transfer. Closes the "PT bypasses
   payroll" gap and the people-cost loop's open question.
5. **Roster generate-and-gate loop** — auto-generate next week's draft
   (schedule-generator is already agentic), run the labour gate, nudge
   manager to review/publish, escalate unpublished by deadline. Extends the
   shipped publish-gate + roster-nudge.
6. **Compliance/cert chase** — the cert cron already writes staged memos;
   compliance reminder flips statuses but notifies nobody. Close delivery +
   acknowledgement tracking + manager escalation. (49 pending events today.)
7. **Missing-profile/rate chase** — the labour gate blocks publish on
   orphans but only at publish time; a standing digest of scheduled staff
   missing profile/rate would have cleared Fatin weeks ago.
8. **Shift-swap + leave SLA chase** — leave-manager agent already
   auto-approves/escalates; swaps have a full state machine; neither chases
   stale `pending_approval`.
9. **Memo acknowledgement chase** — `api/hr/memos` already computes
   `unacknowledged`; nudge + escalate on major-severity memos.
10. **Review-penalty → payroll enforcement** (rule-6) — attribution loop
    exists (sync + EOM digest); the applied penalty still never deducts the
    performance allowance in the run.

Cross-cutting: **none of the ~11 HR crons has a heartbeat** — whatever loop
ships first should carry the watchdog pattern with it (the procurement
loop-watchdog is the template).

## 5. Access-control + correctness sweep (all 68 backoffice HR routes + staff-side HR)

**Clean bill on the big fears.** The two open staff-access-audit issues do
NOT extend into HR: every HR route imports the audience-enforcing
`getSession` from `@/lib/auth` (H-1 clean); every payroll/salary/bank route
is OWNER/ADMIN-only (no MANAGER/STAFF reach); no unauthenticated routes;
MANAGER-permitted routes all apply subtree (`resolveVisibleUserIds`) or
outlet (`canAccessOutlet`) scoping correctly; staff-side HR routes all
self-scope to `session.id`. Privilege-escalation guards on
`employees/[id]/access` (last-owner lockout, field allow-list) are well
built. Payroll item edits refuse confirmed/paid runs. No TODO/FIXME/HACK
markers in HR code.

**Real findings:**

- **FIXED in this PR — `schedules/assign` destructive replace.** Used
  DELETE-then-INSERT (`assign/route.ts:102-123`): a failed insert destroyed
  the existing shift — the exact vanishing-roster failure mode from the
  AI-Fill week-wipe lesson. Its sibling `schedules/cell` was already
  rewritten insert-first; `assign` never got the fix. Now mirrors `cell`.
- **FIXED in this PR — weekly payroll state-machine guards.** `confirm` was
  an unconditional update (could downgrade a `paid` run back to `confirmed`,
  desyncing generated bank files) and `mark_paid` had no precondition (a
  `draft` could jump straight to `paid`, skipping human sign-off). Now
  mirrors the monthly route's atomic `.in("status",…)` + 409 pattern.
- **OPEN — `employees` POST/PUT mass-assignment** (`employees/route.ts:168-199`):
  `req.json()` spread straight into `hr_employee_profiles` with no
  allow-list/zod — that table holds `basic_salary`, `hourly_rate`, EPF/PCB
  numbers, allowance amounts. OWNER/ADMIN-only so not an access breach, but
  it's the one unvalidated writer on the pay-rate path (contrast
  `salary-history`, `payroll/items/[item_id]`, `recurring-items`, which all
  validate). Fix = field allow-list + numeric guards, ~an hour, needs care
  not to break the employee editor's field set.
- **OPEN (low) — `payroll-items` catalog mass-assigns body** (same pattern,
  item definitions only).
- **OPEN (low) — `loe-import/commit`** writes `salary_type:"monthly"` which
  the salary-history API's own enum rejects (`base|hourly|allowance|bonus|
  increment`) so imported salary rows never mirror to `basic_salary`; its
  rollback also doesn't cover the salary/job-history inserts.
- **INFO — two parallel manager-scoping mechanisms** (reporting-line subtree
  vs outlet membership) applied per-domain; internally consistent today but
  they can diverge — fold into the future `withAuth` durable fix.
